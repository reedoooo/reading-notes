Reading: NoSQL vs SQL

  • The best fit for a complex query intensive environment is a SQL database.
  • The best fit for hierarchical data storage is a NoSQL database.
  • Differences in scalability between a SQL and NoSQL database:
    • SQL databases require vertical or horizontal scaling, while NoSQL databases are designed for horizontal scalability from the beginning.
    • Scaling a SQL database can be more challenging and may require complex techniques like sharding or partitioning.
    • NoSQL databases can easily distribute data across multiple servers, allowing for seamless scaling by adding more servers to the cluster.

SQL Modeling Techniques

  • A one-to-many relationship in data tables represents a relationship where one record in a table is associated with multiple records in another table. We relate them by using a foreign key in the "many" table that references the primary key of the corresponding record in the "one" table.
  • Prior to designing a relational database, it might be useful to create a visual diagram, such as an entity-relationship diagram (ERD), of the database tables and their relationships.
  • A primary key is a unique identifier for each record in a table, while a foreign key is a field in a table that refers to the primary key of another table. Primary keys ensure data integrity and uniqueness, while foreign keys establish relationships between tables.

Videos: SQL vs NoSQL

  • In SQL syntax, keywords are reserved words used for specific operations, while parameters are values passed into SQL statements at runtime.
  • Normalization, within the context of schemas and data, refers to the process of organizing data in a database to reduce redundancy and dependency.
  • One-to-one relationship: Each record in one table is associated with exactly one record in another table.
  • One-to-many relationship: Each record in one table is associated with multiple records in another table.
  • Many-to-many relationship: Multiple records in one table are associated with multiple records in another table, requiring a junction table to represent the relationship.
